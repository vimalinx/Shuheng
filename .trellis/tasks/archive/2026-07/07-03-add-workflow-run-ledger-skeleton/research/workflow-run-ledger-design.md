# Workflow Run Ledger Design Notes

## Repository Constraints

- Workflow definitions already load through manifest-declared `PluginWorkflow` records and pure parser helpers.
- `workflows.py` is intentionally pure: it should not import `app.py`, curses, runtime dispatch, approvals, task/progress ledgers, artifacts, provider adapters, subprocess, or governance owners.
- Low-level JSONL behavior already exists in `ledger_store.py`. App-level code can choose the concrete harness path and append the pure row.
- The harness architecture baseline prioritizes explicit ledgers, artifact refs, human approval gates, auditable communication, strong Orchestrator responsibility, and single-writer enforcement.

## MVP Contract

- `run_id`: generated by the app/Orchestrator layer.
- `schema_version`: `shuheng.workflow_run.v1`.
- `status`: `planned`.
- `workflow_ref`, `plugin_id`, `workflow_id`, `workflow_name`, `workflow_path`.
- `inputs`: captured snapshot for future runner work. The MVP command uses `{}`.
- `permissions`: copied metadata from the workflow definition. It does not grant runtime permissions.
- `steps`: ordered snapshots with each step initially `pending`.
- `execution`: explicit no-op flags, e.g. no steps executed, no dispatches, no approvals, no tools.

## Recommended Approach

Use a pure row builder in `workflows.py` and an app-owned append wrapper:

- `workflows.build_workflow_run_record(...)` builds the deterministic row.
- `app.create_planned_workflow_run(...)` chooses `run_id`, timestamp, and `AGENT_WORKFLOW_RUNS_PATH`, then appends via `ledger_store.append_jsonl`.
- `handle_workflow_command(...)` handles `/workflow run <ref>` by validating first, then appending exactly one run row.

## Non-Goals

- No runner state machine beyond `planned`.
- No approval rows.
- No task/progress rows.
- No artifacts.
- No runtime dispatch.
- No plugin code execution.
